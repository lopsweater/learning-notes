# PS vs CS 速度差距·引擎内部手册
（只讲工程真实结论，不讲虚的，可直接用于架构决策）

## 一、核心结论总纲
1. **凡是大量纹理采样 → PS 完胜 CS（快 3~10 倍）**
2. **凡是随机读写、共享内存、排序、聚类 → CS 完胜 PS**
3. **凡是带混合/深度/模板/MSAA → PS 硬件级碾压**
4. **凡是可异步、可和图形并行 → CS 价值不在速度，在“吞吐总量”**

# 二、按任务类型：谁更快？快多少？为什么？

## 1）模糊 / 滤波 / 卷积类（Bloom、高斯模糊、阴影滤波、AO）
- **PS：快**
- **CS：慢 3~8 倍**
- **差距原因**
  - PS 在 **Tile 缓存** 内读写，带宽占用极低
  - PS 走 **专用 TMU 采样路径**，延迟更低、吞吐更高
  - PS 直接写 RT，不需要 UAV 全局内存同步
- **结论**：后处理能写 PS 绝不写 CS

## 2）简单后处理（ToneMapping、FXAA、LUT、ColorGrade）
- **PS：明显快**
- **CS：慢 1.5~3 倍**
- **差距原因**
  - 采样密集 + 片上缓存友好
  - CS 多一步 UAV 写回与缓存刷新
- **结论**：全屏简单效果一律 PS

## 3）粒子渲染（小四边形、大量透明、混合）
- **PS：极快**
- **CS：慢 5~20 倍**
- **差距原因**
  - **Early-Z + 硬件混合** 是降维打击
  - CS 做透明必须原子或复杂排序，完全没法比
- **结论**：粒子渲染必须用 GS+VS+PS 管线

## 4）不透明物体渲染
- **PS：无敌快**
- **CS：无法替代**
- **差距原因**
  - 硬件光栅化 + HiZ / Early-Z 可剔除 70%+ 无效像素
  - 三角形遍历、插值、覆盖都是固化硬件
- **结论**：主渲染不可能用 CS 替代

## 5）光线 marching / 复杂采样（SSR、光线追踪阴影）
- **PS：略快或持平**
- **CS：接近但略慢**
- **差距原因**
  - 依然是采样密集型
  - 但逻辑复杂后，优势缩小
- **结论**：追求极致速度用 PS；追求结构灵活用 CS

## 6）物理/粒子更新（位置、速度、生命周期）
- **CS：极快**
- **PS：极慢甚至做不了**
- **差距原因**
  - CS 有 **共享内存(Shared Memory)**
  - CS 支持高效原子、线程组同步、连续缓冲区访问
  - PS 不能高效写回结构化数据，也不能做真正并行计算
- **结论**：所有“更新类”逻辑必须 CS

## 7）视锥剔除 / 聚类 / 分桶 / 生成 IndirectArgs
- **CS：秒杀 PS**
- **PS：完全不适合**
- **差距原因**
  - 需要紧凑写入、原子计数、前缀和、随机访问
- **结论**：这类任务只能 CS

## 8）贴图生成、高度图更新、纹理拷贝压缩
- **CS：更快**
- **PS：较慢**
- **差距原因**
  - 非采样密集，而是**写密集 / 逻辑密集**
  - UAV 更适合整块、规整、非瓦片式写入

## 9）插值、皮肤、复杂光照（大量纹理+常量）
- **PS：略快**
- **CS：接近**
- **差距原因**
  - 采样仍占优势，但逻辑变重后差距变小
- **结论**：材质主通道依然 PS 更稳

# 三、最精炼“一秒判断法”（引擎通用）
```
如果任务满足下面任意一条 → 优先 PS
  1. 全屏/矩形，采样多、写一次
  2. 需要混合、深度、模板、MSAA
  3. 三角形光栅化相关
  4. 滤波、模糊、抗锯齿、后处理

如果任务满足下面任意一条 → 必须 CS
  1. 要更新结构化数据（粒子、物理、动画）
  2. 要共享内存、原子、排序、聚类、计数
  3. 可异步，希望和图形并行跑提高总吞吐
  4. 非图形输出，只是计算/生成数据
```

# 四、你最关心的一句话总结
- **同样是“采样滤波”这种像素级工作：
  PS 因为 Tile 缓存 + 专用 TMU + 无UAV开销，天然比 CS 快很多。**
- **CS 的优势不是“同任务更快”，
  而是“能让 GPU 同时干别的，总吞吐量更大”。**
